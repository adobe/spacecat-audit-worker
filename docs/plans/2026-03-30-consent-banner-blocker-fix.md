# Fix Paid-Cookie-Consent Audit Pipeline

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Fix two bugs in the paid-cookie-consent audit pipeline so that high-severity consent banner issues create opportunities with suggestions, and null/empty guidance no longer crashes the handler.

**Architecture:** Two repos need changes. Mystique must populate the `suggestions` field in its SQS payload when the crew generates high-severity guidance with screenshots. The audit-worker must handle null/empty `guidance` gracefully instead of crashing. Both fixes are deployed independently — the audit-worker fix is backward-compatible and can ship first.

**Tech Stack:** Node.js (audit-worker, Mocha/Chai/Sinon tests), Python (Mystique, pytest)

---

## Accepted Risks & Notes

- **Deployment gap blackout:** Between audit-worker and Mystique deploys, zero paid-cookie-consent opportunities will be created (same as current broken state). This is acceptable since zero are being created today anyway.
- **Missing `issueSeverity` field:** Sites whose Mystique response lacks `issueSeverity` will no longer create opportunities (previously they did). Accepted — these are edge cases from malformed crew output and should not produce opportunities.
- **No DLQ concern:** The current 178 crashes are caught by the outer try/catch in `src/index.js` (line 327). Messages are consumed on failure, not sent to DLQ. No replay needed after deployment.
- **`skip_suggestions = True` vs payload `suggestions`:** The `skip_suggestions` property on `GeneratePaidConsentGuidanceTask` controls Mystique's internal suggestion generation pipeline (disabled). The `suggestions` field in the SQS payload is a separate signal to the audit-worker that screenshots were generated by the crew. These are independent — `skip_suggestions` stays `True`, and the `get_payload` override populates the payload field based on crew output. A code comment documents this distinction.
- **100% test coverage required** for all `src/**/*.js` files in audit-worker. Test files (`*.test.js`) do not require coverage themselves.

## Task 2: Mystique — Populate `suggestions` in payload for high severity

**Files:**
- Modify: `/tmp/mystique-fix/app/tasks/generate_paid_consent_guidance_task.py`
- Test: `/tmp/mystique-fix/app/tests/tasks/test_generate_paid_consent_guidance_task_payload.py` (new)

Currently `get_payload()` in the base task hardcodes `"suggestions": []`. For paid-cookie-consent, the crew generates high-severity guidance with screenshot data in `body.data`. We need to signal to the audit-worker that suggestions/screenshots are available.

**IMPORTANT:** `GuidanceWithBody.body` is typed as `dict` (not a Pydantic model). All access to keys inside `body` and its nested `data` dict MUST use `.get()`, NOT `getattr()`. `getattr()` on a dict returns `None` for key names — it does not look up dict keys.

```python
from agents.crews.guidance.paid.consent_banner.consent_banner import (
    CookieConsentBannerCrew,
)
from models.opportunities import OpportunityTypeEnum, Opportunity
from tasks.generate_guidance_basetask import GenerateGuidanceBaseTask


class GeneratePaidConsentGuidanceTask(GenerateGuidanceBaseTask):
    def get_crew(self):
        return CookieConsentBannerCrew(base_path="../.tmp")

    @staticmethod
    def get_supported_opportunity_types():
        return [OpportunityTypeEnum.paid_cookie_consent]

    def get_max_guidance_len(self):
        return 1

    def skip_suggestions(self, opportunity):
        # Note: skip_suggestions controls Mystique's internal suggestion generation
        # pipeline (disabled for this task). The 'suggestions' field in the SQS
        # payload is a separate signal populated by get_payload() below, telling
        # the audit-worker that the crew generated screenshots for high severity.
        return True

    def get_payload(self, opportunity: Opportunity):
        payload = super().get_payload(opportunity)

        # Populate suggestions when guidance has high severity with screenshot data.
        # The audit-worker gates opportunity creation on non-empty suggestions
        # (isNonEmptyArray check in guidance-handler.js).
        # payload.data is a mutable dict built by the base class get_payload().
        if opportunity.guidance and len(opportunity.guidance) > 0:
            # guidance[0] is a GuidanceWithBody Pydantic model; .body is a dict
            body = getattr(opportunity.guidance[0], 'body', None)
            if isinstance(body, dict):
                severity = body.get('issueSeverity')
                data = body.get('data')
                if severity and 'high' in str(severity).lower() and isinstance(data, dict):
                    has_mobile = data.get('mobile')
                    has_desktop = data.get('desktop')
                    if has_mobile or has_desktop:
                        payload.data['suggestions'] = [{'type': 'CONTENT_UPDATE'}]

        return payload
```
